No close quote?

CSS quotes & the depth counter.

The problem

In the video, Kevin added a visually appealing (read: purely ornamental) curly quote to a <blockquote> element. Here’s the important bit:

View CSS code.

Doing the above will increase the depth counter. The counter’s value determines which pair of open/close quotation marks from the quotes declaration to render. This is meant to affect nested items, but a stray declaration that isn’t closed also affects subsequent items.

Prevent issues on subsequent items

Assuming our design doesn’t require an ornamental close quote, we could specify close-quote in the after pseudo element and hide it, keeping the designer and the depth counter happy.

Luckily we don’t need to, this is where no-close-quote comes in. By adding it as the content of an after pseudo element, the depth counter that got increased by our use of open-quote goes down by one again, but nothing is shown on screen. Now subsequent elements that also make use of quotes will start at the proper depth level.

View CSS code.

But! If you happen to have an inline quotation <q> element nested inside your blockquote, the depth counter is still up by one. So you end up with different quotation marks on those, which may or may not be what you want.

Tackle nested items

In the past my approach to avoid tampering with the depth counter has been to simply specify the actual character instead of open-quote in the content declaration. However, doing so won’t give you language-aware quotation marks. If your ornamental quote should anyhow look the same everywhere, fine.
