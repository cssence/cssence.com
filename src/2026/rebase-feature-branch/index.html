Rebase feature branch

Extra
2026-03-04T09:55:00Z

When I write a blog post on this site, the workflow to get it live is quite simple: I commit the post on my main branch—aptly named “main”—and push it to GitHub. At this point Netlify takes over and publishes it. Then I let people on the social networks know that I wrote a thing. It’s quite likely that I get some comments this way, those then also become part of the blog post. To get the comments live, in most cases I modify the latest commit with git commit --amend, then I force-push the changes to make sure GitHub also has the modified commit. (I know, I know, but this is a pet project with just one maintainer.)

When I work on a new feature or a blog post that takes a long time to write, I create feature branches, meaning on the main branch I’ll do a git checkout -b feat/xyz. However, if I created such a branch based on a commit that then got modified in the way described above, the branches will run out of sync.

What follows is my take on getting out of this mess. To recap, the feature branch already contains a new commit, but the one before got amended on the main branch.

Getting it right again is a two-step process. On the feature branch, I remove the commit that is no longer found on the main branch. When there is only one feature commit, I’ll do an interactive rebase atop HEAD~2, where I then drop the outdated commit, but make sure to keep the new feature.

If I run into conflicts I need to address those, but in that particular kind of workflow there usually aren’t any.

Getting the missing commit from the main branch back just takes another rebase, this time it’s not interactive.

In this state I could switch to the main branch and merge the feature branch.
